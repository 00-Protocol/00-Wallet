1. Overview

The Unified Onion Relay is a single process that runs two services: a Pubkey Indexer (stealth address scanning via HTTP) and a Joiner Coordinator (blind CoinJoin round management via Nostr). It replaces the v1 HTLC onion routing system with a fundamentally simpler and more private design.

Key Properties:
  • Single process, dual service — pubkey indexer + CoinJoin coordinator in one binary
  • Nostr as transport — relay connects to public Nostr relays as a client, never runs a server (never sees user IPs). Operators can optionally bundle a Nostr relay in the package for full independence from public infrastructure.
  • Blind coordination — relay counts encrypted blobs, triggers rounds, forwards messages. It sees ephemeral pubkeys but not real identities, inputs, outputs, or amounts.
  • Onion-layered outputs — each participant wraps their output address in layers of encryption. The relay peels one layer and forwards. No participant learns another's output address.
  • Ephemeral identity — new Nostr keypair per round, destroyed after completion

2. Architecture

The relay is a single Node.js process running relay.js. It imports pubkey-indexer.js for the indexer service and uses Nostr WebSockets for the Joiner coordinator. Encryption is implemented locally in relay.js (Node.js crypto) and in onion-crypto.js (browser clients). Both use identical algorithms (NIP-44, NIP-59, AES-256-GCM) with platform-appropriate APIs.

3. Nostr Event Kinds

KindNameTagContent
22230 Relay Announcement ['t', '0penw0rld-joiner'] JSON: {services, version, min_participants, max_participants, indexer_url}
Published every 5 minutes. Wallets subscribe to discover available relays.
22231 Joiner Communication ['p', target_pub] NIP-59 gift-wrapped (kind 1059 outer, kind 22231 inner rumor). Triple-layer: rumor → seal → wrap. All Joiner messages: ready signals, round_start, inputs, onion-wrapped outputs, tx skeleton, signatures.
Two kinds only. The v1 system used kind 22232 for settlement notifications. v2 eliminates settlement monitoring entirely — there is only one CoinJoin transaction, and all participants sign it directly.

4. Relay Discovery

Wallets subscribe to kind 22230 events tagged ['t', '0penw0rld-joiner'] on public Nostr relays. Each relay announcement contains the relay's pubkey, supported services, and configuration.

// Wallet subscribes to relay announcements
REQ filter: {
  kinds: [22230],
  '#t': ['0penw0rld-joiner'],
  since: now - 3600
}

// Relay announces itself (every 5 minutes)
announceRelay() → kind 22230 event:
{
  content: JSON.stringify({
    services: ['joiner', 'indexer'],
    version: '2.0.0',
    min_participants: 3,
    max_participants: 10,
    indexer_url: 'http://localhost:3847'
  }),
  tags: [['t', '0penw0rld-joiner']],
  pubkey: relay_xonly_pub
}

// Wallet picks freshest relay, stores relay list
// Relay pubkey is used as NIP-44 encryption target

5. Onion Encryption

Shared module onion-crypto.js is used by wallet clients, the relay, and the Joiner UI. It provides onion layer encryption (secp256k1 ECDH + AES-256-GCM) and NIP-44 authenticated message encryption (ChaCha20 + HKDF + HMAC-SHA256).

Single Layer — onionLayer(data, peelerPubHex)

1. eph = random(32)                         // ephemeral key per layer
2. ephPub = secp256k1.getPublicKey(eph)      // 33 bytes compressed
3. shared = ECDH(eph, peelerPub)[1:33]       // x-coordinate only
4. aesKey = SHA256(shared)                   // 32-byte AES key
5. iv = random(12)                           // 12-byte GCM nonce
6. ct = AES-256-GCM.encrypt(aesKey, iv, data)

Output: ephPub(33) || iv(12) || ciphertext+tag

Peel Layer — onionPeel(blob, myPriv)

1. Extract ephPub (33 bytes) from blob
2. ECDH(myPriv, ephPub) → shared secret
3. aesKey = SHA256(shared)
4. Extract iv (12 bytes)
5. Decrypt remainder with AES-256-GCM
6. Return plaintext

Joiner Output Wrapping — onionWrap(payload, peelerPubHexes)

// Payload format: "addr|value_sats" (e.g. "bitcoincash:qz...abc|50000")
// Padded to JOINER_PAD_SIZE = 80 bytes with 0x01 delimiter

1. raw = utf8(payload)
2. padded = new Uint8Array(80)
3. padded.set(raw); padded[raw.length] = 0x01
4. For each peeler pub (last to first):
     padded = onionLayer(padded, peelerPub)
5. Return nested encrypted blob

// Each participant in the round is a "peeler"
// The relay peels the outer layer, participants peel the rest
// Final output address is only revealed inside the CoinJoin construction

Unpad — onionUnpad(data)

// After all layers are peeled:
1. Find 0x01 delimiter
2. Decode UTF-8 up to delimiter
3. Parse "addr|value" → { addr, value }
// Returns the output address and sats amount for the CoinJoin tx

NIP-44 — nip44Encrypt / nip44Decrypt

// Used for all kind 22231 Joiner communication
// NIP-44 v2: ChaCha20 + HKDF-SHA256 + HMAC-SHA256

// 1. Conversation key (computed once per key pair)
shared_x         = ECDH(myPriv, theirPub).x              // 32-byte x-coordinate
conversation_key = HKDF-extract(SHA256, shared_x, "nip44-v2")  // 32 bytes

// 2. Per-message keys (HKDF-expand with random nonce)
nonce(32)    = random()
keys(76)     = HKDF-expand(SHA256, conversation_key, nonce, 76)
  chacha_key   = keys[0:32]     // ChaCha20 key
  chacha_nonce = keys[32:44]    // ChaCha20 nonce (12 bytes)
  hmac_key     = keys[44:76]    // HMAC-SHA256 key

// 3. Encrypt: pad → ChaCha20 → HMAC
padded     = [length_BE(2)] + plaintext + zero_padding_to_power_of_2
ciphertext = ChaCha20(chacha_key, chacha_nonce, padded)
mac        = HMAC-SHA256(hmac_key, nonce || ciphertext)

// 4. Wire format: base64( 0x02 || nonce(32) || ciphertext || mac(32) )

6. Ready Signal & Round Triggering

  1. Ephemeral key — User generates a fresh Nostr keypair for this round
  2. Send ready — NIP-59 gift-wrapped message to relay pubkey (kind 1059, inner kind 22231): 
    {
  type: 'ready',
  utxo_count: 3,    // number of inputs the user will contribute
  output_count: 2   // number of outputs the user wants
}
  3. Relay receives — Decrypts the outer NIP-44 layer. Sees the ephemeral pubkey + metadata (utxo_count, output_count). Adds to ready queue.
  4. Threshold check — Every 5 seconds, relay prunes stale entries (older than 30s window) and checks if >= minParticipants (default 3) are queued
  5. Round trigger — When threshold is met, relay takes up to maxParticipants (default 10) from the queue and starts a round
No public Nostr event = no fingerprint. Ready signals are NIP-59 gift-wrapped — Nostr relays see only a disposable ephemeral pubkey, a randomized timestamp, and encrypted content. They cannot identify who is ready or when rounds are forming.

7. Round Coordination

Once the relay has enough ready participants, it starts a round:

  1. Generate round_id — random(16) hex string
  2. Shuffle mix_order — Fisher-Yates shuffle of participant ephemeral pubkeys. This determines the onion peeling chain.
  3. Send round_start — NIP-44 encrypted to each participant: 
    {
  type: 'round_start',
  round_id: 'a1b2c3...',
  participants: ['pub1', 'pub2', 'pub3', ...],
  mix_order: ['pub3', 'pub1', 'pub2', ...]   // shuffled peeling order
}
  4. Message bus — Relay receives onion-wrapped blobs and round messages from participants. It peels the outer NIP-44 layer, then:
    • If message has a target field: forward only to that specific participant
    • Otherwise: broadcast to all round participants except sender
  5. Round timeout — 360 seconds total per round (60s × 6 phases). Non-responders cause the round to be deleted. Participants can re-queue with new ephemeral keys.
// Round lifecycle:
//
// Phase 1: READY      — users send ready signals (pre-round)
// Phase 2: ROUND_START — relay sends round_start to all
// Phase 3: INPUTS     — participants announce their UTXOs
// Phase 4: OUTPUTS    — participants send onion-wrapped output addresses
// Phase 5: TX BUILD   — coordinator (or participant) builds unsigned tx
// Phase 6: SIGN       — each participant signs their inputs
// Phase 7: BROADCAST  — signed CoinJoin tx broadcast to network

8. Ephemeral Identity

Each participant generates a new Nostr keypair for every round. The ephemeral private key is used to NIP-44 encrypt/decrypt messages with the relay and other participants. It is destroyed after the round completes (success or failure).

Properties:
  • Unlinkable across rounds — The relay sees ephemeral pubkeys but cannot correlate them to real identities or to previous rounds
  • No persistent identity — There is no "account" or "session" on the relay. Each round is a fresh anonymous interaction.
  • Ban resistance — A griefer who disrupts a round simply gets a new identity in the next round. Ban lists are pointless against ephemeral keys. Instead, the protocol relies on economic cost (UTXOs must be real) and phase timeouts.
  • Forward secrecy — Compromising the relay's private key after a round does not reveal past participants, because the ephemeral keys that encrypted messages to the relay are already destroyed.

9. Pubkey Indexer Integration

The same relay process serves an HTTP API for stealth address scanning. It connects to Fulcrum (WSS) or BCHN (RPC) to extract public keys from transaction inputs across block ranges.

// Client requests all pubkeys for a block range:
GET /api/pubkeys?from=830000&to=830100

// Response: array of {txid, pubkeys} for every input in that range
// Client downloads ALL pubkeys, then filters locally with their scan key
// Server doesn't know what the client is looking for
Privacy by design: The indexer returns all pubkeys for the requested range indiscriminately. The client performs ECDH locally with their scan private key to detect payments. The server has no knowledge of which pubkeys belong to the client or which transactions are relevant.

11. Security & Privacy

What the relay sees

DataVisible?Notes
Ephemeral pubkeysYesOne-time keys, destroyed after round
Blob count per roundYesKnows how many participants, not who they are
Timing of ready signalsYesKnows when participants join, not their identity
UTXO count / output countYesMetadata from ready signal

What the relay does NOT see

DataVisible?Why
Real identitiesNoOnly ephemeral pubkeys, fresh per round
Input UTXOsNoParticipants exchange inputs directly via onion encryption
Output addressesNoOnion-wrapped with multi-layer encryption
AmountsNoEncoded inside onion payloads
User IPsNoRelay connects to Nostr as a client, not a server

What Nostr relays see

DataVisible?Notes
User IPsYesMitigated by Tor option or by running a bundled Nostr relay in the onion relay package
Encrypted blobsYesNIP-44 ciphertext, cannot decrypt without private keys
Message contentNoAll payloads are NIP-44 authenticated encryption end-to-end

Comparison with CashFusion server

CashFusion server sees everything: The CF server receives all inputs and all outputs in plaintext during round construction. It knows which participant contributed which inputs and requested which outputs. A compromised CF server can fully deanonymize every round. The 00 Unified Relay is blind — it coordinates timing and forwarding without ever seeing inputs, outputs, or amounts.

13. Comparison with v1 (HTLC Onion)

Propertyv1 — HTLC Onion Routingv2 — Unified Relay (CoinJoin)
Transaction count 6+ tx per payment (fund + claim per hop) 1 tx — single CoinJoin transaction
On-chain footprint HTLC scripts (OP_SHA256 + OP_CLTV) visible on-chain Standard P2PKH inputs/outputs, indistinguishable from normal tx
Settlement Preimage-based claim chain, settlement monitoring loop All participants sign once, single broadcast
Relay knowledge Each hop sees previous + next hop, amounts, and HTLC hashes Relay sees ephemeral pubkeys and blob count only
Failure mode HTLC timeout refunds (hours). Race conditions between claim and refund. Round aborts instantly. No funds at risk. Re-queue with new key.
Fee structure Per-hop relay fees + miner fees for each HTLC tx Single tx miner fee split among participants. No relay fee.
Complexity HTLC scripts, hash decoration, CLTV chains, settlement monitoring One CoinJoin tx, onion-wrapped outputs, NIP-44 authenticated encryption
Nostr events 3 kinds (22230, 22231, 22232) 2 kinds (22230, 22231)
Note: HTLC-based onion routing has been moved to a separate multichain atomic swap mixer project. The 00 wallet now uses the Unified Relay exclusively for privacy mixing.
