1. Overview

00 Joiner implements a serverless CoinJoin coordinated through a blind onion relay. All participant communication is onion-encrypted so the relay cannot read message contents. Ephemeral keys are generated per round, preventing identity persistence across mixes. Small rounds are silent by default — they look like normal multi-input payments on-chain.

Key Properties:
  • Blind relay — relay forwards encrypted blobs it cannot decrypt
  • Onion-encrypted communication — all phases wrapped in layered encryption
  • Ephemeral keys per round — no identity persistence, unlinkable across rounds
  • Variable participants — 3 to 10+ peers per round
  • Silent by default — small rounds indistinguishable from normal payments
  • Auto-mix — 3 pipelined rounds complete in under 30 seconds after receiving funds
  • Pipelining — outputs from round N feed directly into round N+1 (no confirmation wait)

2. Protocol Flow (6 Phases)

[Phase 1: READY SIGNAL]
    Participant encrypts ready blob to relay pubkey
    Relay decrypts, counts ephemeral pubkeys
    When 3+ ready in 30s window -> round_start
         |
[Phase 2: INPUT REGISTRATION] (60s timeout)
    Each peer: select 1-5 UTXOs randomly
    Declare output count (2-4)
    Send onion-wrapped inputs via relay
    Relay forwards encrypted blobs (cannot read)
         |
[Phase 3: BLIND OUTPUT] (60s timeout)
    Each peer: derive stealth addresses for outputs
    Wrap each output in N onion layers (one per participant)
    Forward through peeler chain via relay
    Each peeler strips a layer -> shuffles -> forwards to next
    Last peeler reveals plaintext outputs
         |
[Phase 4: TX ASSEMBLY]
    Last peeler assembles unsigned TX
    All inputs + all blind outputs + fee
    Broadcasts unsigned TX via relay
         |
[Phase 5: VERIFY & SIGN] (60s timeout)
    Each peer verifies:
      - All own outputs present
      - No inflation (totalOut <= totalIn)
      - Fees reasonable (< 5 sat/byte)
    Sign each own input (Schnorr, lowS, SIGHASH_ALL|FORKID)
    Send onion-wrapped signatures via relay
         |
[Phase 6: BROADCAST & PIPELINE]
    First peer to collect all sigs assembles final TX
    Broadcast to BCH network via Fulcrum
    If pipelining: outputs -> next round immediately
    DONE (or next round)

3. Nostr Communication

The protocol uses only 2 Nostr event kinds. The old kinds 22222, 22223, and 22224 are removed. No public pool events means no on-chain or relay-level fingerprint for mixing activity.

KindNameEncryptionPurpose
22230RELAY ANNOUNCEMENTNone (public)Relay publishes available services and its pubkey
1059GIFT WRAPNIP-59 (NIP-44 double-layer)ALL communication: ready, round_start, inputs, onions, outputs, tx, signatures. Inner rumor kind 22231.
Privacy gain: All messages are NIP-59 gift-wrapped (kind 1059). Nostr relays see only disposable ephemeral pubkeys and randomized timestamps — they cannot identify the sender, the recipient's real identity, or correlate messages by timing. The inner rumor (kind 22231) is double-encrypted through seal + wrap layers.

4. Ephemeral Round Identity

Each round generates a completely fresh identity via _generateRoundIdentity(). The ephemeral key pair is used for all communication during that round, then destroyed.

// Ephemeral identity lifecycle
function _generateRoundIdentity() {
  const priv = crypto.getRandomValues(new Uint8Array(32))
  const pub  = secp256k1.getPublicKey(priv, true)  // compressed
  return { priv, pub }
}

// Used for:
//   - Ready signal to relay
//   - All round communication (kind 22231)
//   - Nostr subscriptions for this round

// After round completes or fails:
function _destroyRoundIdentity() {
  roundKey.priv.fill(0)   // zero out private key
  roundKey = null
}
Implications:
  • Cannot link participant across rounds — new key pair each time
  • Ban list removed — pointless with ephemeral keys (griefer gets a fresh identity anyway)
  • Relay sees ephemeral pubkeys only — no persistent identity to correlate

5. Ready Signal

There is no pool discovery phase. Instead, a participant privately signals readiness to the relay.

// Ready signal (encrypted to relay pubkey)
payload = {
  type:         "ready",
  utxo_count:   3,        // how many inputs the participant has
  output_count: 2         // how many outputs they want
}

// Encrypted via NIP-44 to relay pubkey
// Relay decrypts, sees: ephemeral pub + metadata
// Relay accumulates ready signals
// When 3+ participants within 30s window:
//   -> relay broadcasts round_start to all ready participants
Silent: The ready signal is a private NIP-44 encrypted message to the relay. It is not a public Nostr event. No external observer can see who is waiting to mix.

6. Phase 2: Input Registration

UTXO Selection

// Select 1-5 random UTXOs
usable   = utxos.filter(u => u.value >= 546)        // exclude dust
nInputs  = random(1, min(5, usable.length))          // random count
selected = cryptoShuffle(usable).slice(0, nInputs)   // random pick

Output Splitting

No change output. All outputs — including what would normally be "change" — are random-amount stealth addresses. This follows the CashFusion combinatorial method: each participant contributes 1-N inputs and receives 1-N outputs with random amounts. There is no distinction between payment and change in a CoinJoin. 
// Split total value into 2-4 stealth outputs
balance  = sum(selected) - myFeeShare
nOutputs = random(2, min(4, floor(balance / 600)))
amounts  = randomSplit(balance, nOutputs)   // random weights, min 546 sats

Relay-Mediated Delivery

Input data is sent via sendViaRelay(): the message is NIP-44 encrypted to the relay pubkey. The relay decrypts the outer envelope, then forwards the inner encrypted payload to all round participants. The relay can see the sender's ephemeral pubkey and the list of recipients, but cannot read the input data itself (inner onion layer).

7. Phase 3: Blind Output (Onion Mix-Net)

Each participant wraps their output addresses in N nested encryption layers — one layer per participant in the mix_order. The onion chain travels through every peeler via the relay. Only the last peeler sees the plaintext addresses.

Payload Format

// Each output: address|value padded to fixed size
payload = pad("bitcoincash:qz...addr" + "|" + "50000", JOINER_PAD_SIZE)
// JOINER_PAD_SIZE = 80 bytes — fixed size prevents length-based deanonymization

Onion Wrapping

// onionWrap() from onion-crypto.js
// Wrap: innermost = last peeler, outermost = first peeler
for peeler in reverse(mix_order):
  ephKey  = randomPrivateKey()
  shared  = ECDH(ephKey.priv, peeler.pub)          // secp256k1 ECDH
  aesKey  = SHA256(shared)
  iv      = crypto.getRandomValues(12)
  cipher  = AES-256-GCM(payload, aesKey, iv)
  payload = ephKey.pub(33) || iv(12) || ciphertext+tag(16)

Peeling Process

First peeler:   receives onions via relay
                -> peels outer layer (ECDH + AES-256-GCM decrypt)
                -> shuffles order
                -> forwards to next peeler via relay

Second peeler:  peels next layer -> shuffles -> forwards via relay

...

Last peeler:    peels final layer -> plaintext outputs revealed
                -> assembles TX with all inputs + revealed outputs

Stealth Addresses

Each output address is derived via _deriveSelfStealth() — a one-time stealth address that prevents address reuse. See Section 13 for the derivation.

8. Phase 4-5: Assembly & Signing

Phase 4: TX Assembly

The last peeler in the mix order constructs the unsigned transaction containing all registered inputs and all revealed blind outputs. The unsigned TX is broadcast to all participants via the relay.

Phase 5: Verify & Sign

// Each participant verifies the unsigned TX:
assert(allOwnOutputsPresent(tx))          // all my stealth outputs included
assert(totalOutputValue <= totalInputValue) // no inflation
assert(feeRate < 5)                        // sat/byte — reasonable fees

// Sign each own input (Schnorr)
for each ownInput:
  script  = P2PKH(hash160(myPub))
  sighash = BIP143(tx, inputIdx, script, value, SIGHASH_ALL | FORKID)
  sig     = secp256k1.sign(sighash, myPriv, { lowS: true })
  scriptSig = push(DER(sig) || 0x41) + push(compressedPub)

// Signatures sent via relay (NIP-44 encrypted kind 22231)

9. Phase 6: Broadcast & Pipeline

The first participant to collect all valid signatures assembles the final signed transaction and broadcasts it to the BCH network via Fulcrum.

Pipelining

When pipelining is enabled, outputs from round N become inputs for round N+1 immediately — no confirmation wait required. BCH allows up to 50 unconfirmed ancestors in the mempool, making deep pipelining possible.

Pipeline Parameters:
  • Safety cap: 10 chained rounds maximum (well within 50-ancestor limit)
  • Throughput: 3 rounds back-to-back in under 30 seconds (default auto-mix target)
  • Per-round timing: ~10s (2s inputs + 3s onion + 1s assembly + 2s signing + 1s broadcast)
  • New identity: fresh ephemeral key pair for each pipelined round

10. Auto-Mix

Auto-mix is the default behavior. After receiving funds, the wallet automatically enters a privacy pipeline of 3 rounds.

// Auto-mix lifecycle (toggleAutoMix() + autoMixNext())
autoMixNext():
  1. Detect new UTXOs (incoming funds)
  2. Signal ready to relay (ephemeral key)
  3. Execute 3 pipelined rounds:
       Round 1: fresh inputs  -> stealth outputs
       Round 2: round 1 outputs -> new stealth outputs
       Round 3: round 2 outputs -> final stealth outputs
  4. Label resulting UTXOs as "mixed"
  5. Stop — wallet ready for spending
Same participants across rounds is fine: Each round uses new ephemeral keys, new stealth addresses, and different output amounts. Even if the same peers mix together in consecutive pipelined rounds, the cryptographic unlinkability holds.

Timing

PhaseDuration
Input registration~2 seconds
Onion routing~3 seconds

Verify & sign~2 seconds

3-round auto-mix< 30 seconds

11. Grief Mitigation

The signing phase (Phase 5) is the natural blame point. If a participant fails to provide valid signatures within the timeout, they are identified and removed.

Grief handling:
  • Non-signer detected — the peer who did not sign is kicked from the round
  • Restart — if 3+ participants remain, the round restarts without the non-signer
  • Abort — if fewer than 3 remain, the round is aborted
  • No ban list — ephemeral keys mean a griefer gets a fresh identity next round anyway
  • Cost to griefer — each grief attempt costs the attacker time for zero gain (no funds at risk, round simply restarts)
No bond, no punishment. The protocol does not attempt to economically penalize griefers. The defense is simplicity and speed — restarting a round costs seconds, not minutes. Persistent griefing is possible but unprofitable.

12. Onion Encryption

All cryptographic primitives are in the shared module onion-crypto.js, reused by both the Joiner and the onion relay.

onionLayer (single layer)

// Ephemeral ECDH + AES-256-GCM
ephPriv   = randomPrivateKey()                 // 32 bytes
ephPub    = secp256k1.getPublicKey(ephPriv)     // 33 bytes compressed
shared    = ECDH(ephPriv, recipientPub)         // secp256k1 shared secret
aesKey    = SHA256(shared)                       // 32-byte AES key
iv        = crypto.getRandomValues(12)           // 96-bit nonce

ciphertext = AES-256-GCM.encrypt(plaintext, aesKey, iv)

// Wire format: ephPub(33) || iv(12) || ciphertext+tag(16)
// Total overhead per layer: 61 bytes

onionWrap (nested layers)

// Wrap payload in N layers for N peers in mix_order
function onionWrap(payload, peerPubs) {
  padded = pad(payload, 80)                    // JOINER_PAD_SIZE = 80
  blob = padded
  for (let i = peerPubs.length - 1; i >= 0; i--) {
    blob = onionLayer(blob, peerPubs[i])       // innermost first
  }
  return blob
}

NIP-59 Gift Wrap (message encapsulation)

// Every Joiner message is triple-layered via NIP-59:

// Layer 1: RUMOR (unsigned, deniable)
rumor = {
  kind: 22231, pubkey: author_ephemeral_pub, content: joiner_payload,
  created_at: now, tags: [], id: hash(...)
  // NO signature — provides deniability
}

// Layer 2: SEAL (kind 13, signed by author's ephemeral key)
seal = {
  kind: 13, pubkey: author_ephemeral_pub,
  content: NIP-44-encrypt(rumor, author_priv, recipient_pub),
  created_at: now - random(0..48h),  // randomized timestamp
  tags: [],  // MUST be empty
  sig: schnorr_sign(author_priv)
}

// Layer 3: GIFT WRAP (kind 1059, signed by disposable key)
wrap = {
  kind: 1059, pubkey: disposable_ephemeral_pub,
  content: NIP-44-encrypt(seal, disposable_priv, recipient_pub),
  created_at: now - random(0..48h),  // randomized timestamp
  tags: [["p", recipient_pub]],      // routing only
  sig: schnorr_sign(disposable_priv)
}

// What Nostr relays see: kind 1059, disposable pubkey, random timestamp
// What Nostr relays DON'T see: sender, content, real timestamp

NIP-44 (inner encryption)

// Used for all relay communication (kind 22231)
// ChaCha20 + HKDF-SHA256 + HMAC-SHA256 (NIP-44 v2)

// 1. Conversation key (HKDF-extract)
shared_x       = ECDH(myPriv, theirPub).x            // 32-byte x-coordinate
conversation_key = HKDF-extract(SHA256, shared_x, "nip44-v2")

// 2. Message keys (HKDF-expand)
nonce          = random(32)                            // per-message nonce
keys(76 bytes) = HKDF-expand(SHA256, conversation_key, nonce, 76)
  chacha_key   = keys[0:32]                            // 32-byte ChaCha20 key
  chacha_nonce = keys[32:44]                           // 12-byte ChaCha20 nonce
  hmac_key     = keys[44:76]                           // 32-byte HMAC key

// 3. Encrypt
padded     = nip44Pad(plaintext)                       // power-of-2 padding
ciphertext = ChaCha20(chacha_key, chacha_nonce, padded)
mac        = HMAC-SHA256(hmac_key, nonce || ciphertext)

// 4. Wire format: base64(version(1) || nonce(32) || ciphertext || mac(32))
nip44Encrypt(myPriv, theirPubHex, msg) -> base64 payload
nip44Decrypt(myPriv, senderPubHex, b64) -> plaintext

13. Stealth Output Derivation

Every fusion output uses a unique one-time stealth address. The derivation is deterministic from the input key and outpoint, allowing recovery via standard stealth scanning.

Not from xpub. The scanPub and spendPub used below are derived from BIP352 hardened branches (m/352'/145'/0'/1' for scan, m/352'/145'/0'/0' for spend), not from an xpub. The Joiner uses these keys directly for self-ECDH — no non-hardened derivation is involved. Even if one branch leaks, the other remains protected. 
// _deriveSelfStealth(inputPriv, outpoint, outputIdx)
function _deriveSelfStealth(inputPriv, outpoint, outputIdx) {
  shared = ECDH(inputPriv, scanPub)                  // self-ECDH
  nonce  = SHA256(outpoint || u32LE(outputIdx))       // deterministic per output
  c      = SHA256(SHA256(shared_x) || nonce) mod N    // tweak scalar
  stealthPub  = spendPub + c * G                      // tweaked pubkey
  stealthPriv = spendPriv + c mod N                   // tweaked privkey
  return { addr: toCashAddr(stealthPub), priv: stealthPriv }
}
Recovery: Stealth fusion outputs are recoverable via the standard stealth scanning mechanism (scan key + blockchain analysis). The indexer API can be used for efficient recovery.

15. Security

Implemented protections:
  • AES-256-GCM onion layers (authenticated encryption per hop)
  • NIP-59 Gift Wrap (triple-layer: rumor + seal + wrap with randomized timestamps)
  • NIP-44 authenticated encryption (ChaCha20 + HMAC-SHA256) at both seal and wrap layers
  • Output verification before signing (presence + no inflation + fee check)
  • Schnorr signatures with lowS normalization
  • Stealth output derivation (no address reuse)
  • Ephemeral round keys (no identity persistence)
  • No public pool events (no fingerprint)
Known limitations:
  • Final peeler sees all plaintext outputs before broadcast (same trade-off as CashFusion shuffler)
  • On-chain output amounts are visible (no confidential transactions on BCH)
  • No sybil resistance — attacker can join with multiple ephemeral keys (same as CashFusion)
  • Relay sees IP addresses of participants (mitigated by Tor)
